# 石论交易决策系统 - 二期实施清单

本文档用于承接《06_二期技术PRD》，把二期目标进一步拆成可执行的实施顺序、任务清单、代码减法原则和阶段产物。

它不是新的 PRD，而是二期落地时的执行清单。

## 1. 实施原则

二期推进遵循四条原则：

### 1.1 策略优先

优先解决因子有效性、量价筹码、卖点持仓、组合执行，不先做大平台。

### 1.2 回测优先

任何新增规则，优先进入本地研究型回测体系验证；不能验证的规则，不进入主链路。

### 1.3 轻量优先

聚宽侧只保留单文件可运行的轻策略核，本地侧只保留研究、训练、回测必需的工程支撑。

### 1.4 代码减法优先

二期不鼓励继续增加新的 `score`、`flag`、`label` 和临时例外分支，而是优先做归并、收敛、删减。

## 2. 代码减法原则

结合 `x-algorithm` 这类复杂系统的实现经验，石论二期在代码结构上重点借鉴的不是“模型更大”，而是“抽象更少、阶段更清、接口更稳”。

石论应明确采用以下减法原则：

### 2.1 只保留一条主策略流水线

统一主链路为：

`universe -> features -> factors -> ranking -> portfolio -> execution/backtest`

以后新增逻辑，原则上只能挂在这条主链路上，不再新增平行评分链或旁路判断链。

### 2.2 只保留少数稳定角色

尽量把实现压缩为以下角色：

- `Source`：股票池和原始数据来源
- `FeatureBuilder`：基础特征计算
- `FactorBuilder`：因子与风险信号生成
- `Filter`：候选过滤
- `Ranker`：排序
- `PortfolioBuilder`：组合与仓位
- `Executor/BacktestAdapter`：回测与执行适配

如果某段代码不能明确归到这几个角色之一，就要先质疑它是否真的有必要存在。

### 2.3 少造新字段，多做字段归并

二期尽量不再直接新增分散的经验字段，而是把现有字段收敛为少数几组核心因子：

- 趋势因子
- 结构因子
- 量价因子
- 筹码因子
- 风险因子
- 环境因子

原则上，新的业务判断优先通过组合现有因子实现，而不是先发明新的 `flag`。

### 2.4 少手工调分，多做因子验证

凡是带有明显经验加减分痕迹的逻辑，都要进入二期清理对象。

优先改造方式：

- 先证明该因子是否有效
- 再决定是否保留
- 保留后进入统一排序或统一风控

### 2.5 少入口，多标准入口

二期只保留以下几类标准入口：

- 单票分析入口
- 全市场快照入口
- 本地研究回测入口
- 聚宽策略导出入口
- 训练与评估入口

避免再产生新的临时脚本入口和临时分析流程。

### 2.6 少文档分叉，多文档分层

文档层只保留以下分工：

- `项目结构中文说明.md`：新人导读
- `docs/trading_system_detailed_spec.md`：详细技术口径
- `06_二期技术PRD.md`：二期正式目标
- `07_二期实施清单.md`：二期落地清单

除这几类文档外，新增说明应尽量归档到上述主文档，而不是继续散落。

## 3. 二期实施主线

二期按五条主线推进：

1. 回测双线建设
2. 策略信号深化
3. 组合与执行闭环
4. 工程支撑与调度
5. 代码减法与结构收敛

## 4. 里程碑拆解

## M1：回测双线搭起来

目标：

- 把“本地研究型回测”和“聚宽轻策略回测”彻底拆清
- 建立统一代码入口
- 明确哪些逻辑在本地，哪些逻辑在聚宽

### P0 任务

- 建立 `research/validation/` 目录，作为因子有效性分析入口
- 建立 `research/backtest_local/` 目录，作为本地组合回测入口
- 建立 `research/reports/` 目录，作为评估报告输出目录
- 建立 `shilun/portfolio/` 目录，承接排序、仓位、组合规则
- 建立 `scripts/export_joinquant_strategy.py`，承接聚宽轻策略导出
- 明确聚宽轻策略只消费“候选、排序、仓位、风控”四类结果，不直接依赖整套工程

### P1 任务

- 梳理 `shilun/backtest/joinquant.py` 中哪些逻辑属于“执行适配”，哪些逻辑属于“策略本体”
- 设计本地研究回测的标准输入格式
- 设计聚宽导出脚本的标准输出格式

### 本阶段减法任务

- 停止继续往聚宽模板里搬运整套石论工程
- 停止新增“临时回测脚本”
- 停止在 `snapshot_job` 中继续累积回测研究逻辑

### 完成标志

- 本地研究回测入口清晰
- 聚宽轻策略导出入口清晰
- 回测双线边界文档定稿

## M2：策略信号深化

目标：

- 从经验评分系统转向因子化表达
- 完成量价、筹码、卖点与持仓规则的第一轮收敛

### P0 任务

- 建立石论因子池清单，按趋势、结构、量价、筹码、风险、环境分组
- 对现有核心字段做有效性梳理，形成“保留/观察/淘汰”结论
- 形成量价深化研究主题清单
- 形成筹码深化研究主题清单
- 形成卖点与持仓规则清单

### P1 任务

- 将 `invalidation_level` 升级为统一风险退出字段
- 明确“假突破失败退出”和“高位分布防守退出”的可回测定义
- 明确市场和板块过滤如何作用于选股、排序和仓位

### 本阶段减法任务

- 暂停继续新增离散 `flag`
- 暂停继续新增手工经验加减分项
- 把重复表达同一含义的字段收敛为统一因子

### 完成标志

- 因子池文档完成
- 第一版量价/筹码/持仓规则完成
- 因子保留与淘汰原则完成

## M3：组合与执行闭环

目标：

- 让石论从单票建议变成轻量组合策略
- 把交易约束和仓位约束真正接进回测与执行

### P0 任务

- 建立基础股票池机制
- 接入 ST、停牌、涨跌停、成交额过滤
- 建立排序器与候选筛选器
- 建立最大持仓数和单票上限控制
- 建立试仓、标准仓、防守仓三档仓位
- 建立环境驱动的整体降仓机制
- 建立交易成本和滑点模型
- 将 `invalidation_level` 接入减仓或止损逻辑

### P1 任务

- 建立调仓频率策略
- 建立卖出优先级规则
- 建立无法成交时的兜底逻辑
- 建立组合回测输出格式

### 本阶段减法任务

- 不再让 `ActionMapper` 承担过多组合职责
- 不再让 `execution_score` 同时承担“解释、排序、仓位、回测”四种职责
- 排序和仓位逻辑从单票解释逻辑中剥离

### 完成标志

- 候选池、排序、仓位、交易约束可运行
- 组合回测能稳定输出结果
- 聚宽轻策略可根据同一套轻量策略核运行

## M4：工程支撑闭环

目标：

- 形成训练、评估、版本、调度、归档的最小闭环
- 明确本地、云端、Java 的职责边界

### P0 任务

- 固化训练脚本入口
- 固化模型评估脚本入口
- 固化因子验证脚本入口
- 固化日报或周报输出入口
- 设计模型版本与特征版本记录方式
- 设计回测报告与训练产物归档方式

### P1 任务

- 设计阿里云侧的定时训练与定时回测方案
- 设计 OSS 或对象存储归档方案
- 设计服务查询接口的部署方式
- 设计 Java 在任务编排、配置管理、审计追踪上的中长期职责

### 本阶段减法任务

- 云端只做调度、归档、服务，不承载研究本体
- Java 只做稳定入口和治理，不提前承接策略实验逻辑
- 本地脚本只保留研究必需入口，不继续横向发散

### 完成标志

- 本地与云端职责边界明确
- 关键任务支持定时执行
- 训练、回测、报告产物可追踪

## M5：代码结构收敛

目标：

- 在不伤害策略表达力的前提下，显著降低代码层复杂度

### P0 任务

- 以“主流水线”为准重新标注现有模块职责
- 把重复职责文件列入合并清单
- 把高耦合评分逻辑列入收敛清单
- 建立“新增字段必须说明归属因子组”的规则
- 建立“新增规则必须说明回测验证路径”的规则

### P1 任务

- 收敛 `snapshot` 字段，只保留对决策、回测、解释真正必要的核心字段
- 收敛报告输出字段，减少展示层的冗余列
- 收敛文档口径，避免同一逻辑在多个文档中分别演化

### 当前优先收敛对象

- `shilun/features/entry_features.py`
  - 问题：量价、位置、风险、买点相关字段过多，容易继续外扩
  - 目标：按“量价因子组”和“风险因子组”重组输出

- `shilun/decision/execution_engine.py`
  - 问题：当前同时承担评分、仓位、风控、部分排序语义
  - 目标：保留执行层职责，把排序和组合职责外移

- `shilun/jobs/snapshot_job.py`
  - 问题：当前已经承载了不少导出、排序、展示职责
  - 目标：保留批处理编排职责，把排序规则和报告规则外移

- `docs/trading_system_detailed_spec.md`
  - 问题：内容很完整，但后续容易和规划文档重复演化
  - 目标：作为“唯一详细技术口径”，避免其它文档复制细节

### 完成标志

- 主流水线职责图定稿
- 重复评分职责完成第一轮收敛
- 新增代码遵循统一角色边界

## 5. 本地、云端、Java 分工

### 本地负责

- 因子研究
- 策略实验
- 本地回测
- 快速调参与复盘
- 单票分析和全市场快照

### 云端负责

- 定时训练
- 定时批量回测
- 报告与模型产物归档
- 对外查询服务部署

### Java 负责

- 稳定任务编排入口
- 配置中心与版本追踪
- 状态监控和审计记录
- 面向外部的稳定服务治理

### 明确不做的事情

- 不把研究主逻辑迁到 Java
- 不把云端做成策略实验场
- 不为了工程整洁而牺牲策略试错效率

## 6. 二期阶段产物

二期至少应形成以下产物：

1. 一份因子池清单
2. 一份因子有效性分析报告
3. 一份量价与筹码优化清单
4. 一份卖点与持仓规则清单
5. 一套本地研究型回测入口
6. 一套聚宽轻策略导出入口
7. 一套组合与仓位控制规则
8. 一份训练与评估入口规范
9. 一份云端调度与归档方案
10. 一份代码收敛清单

## 7. 二期执行顺序建议

建议按以下顺序执行：

1. 先做 M1，把回测双线和目录边界拆清
2. 再做 M2，把因子、量价、筹码、卖点规则沉淀下来
3. 然后做 M3，把组合、仓位、交易约束接起来
4. 再做 M4，把训练、调度、归档补成闭环
5. 全程并行推进 M5，持续做代码减法，而不是最后再收拾

## 8. 一句话执行要求

二期所有新增代码都要先回答三个问题：

1. 它属于哪一个稳定角色
2. 它能被哪条回测链验证
3. 它是否让系统更简单，而不是更绕
