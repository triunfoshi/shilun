# 石论交易决策系统 - 三期迭代

本文档承接《07_二期实施清单》，用于整理 Sequoia-X、BaoStock 交易技术商城、Shadowell/Alpha 三个外部来源对石论后续演进的借鉴，并把三期目标收敛成可执行的迭代清单。

三期不是继续扩大单票分析链路，而是把石论从“能批量输出排名”推进到：

> 有候选池、有规则中心、有验证门禁、有轻量产品闭环的策略迭代系统。

## 1. 三期背景

二期已经完成了几个重要基础：

- `ShilunPipeline` 成为单票分析主入口
- `SnapshotJob` 能做全市场快照、排名、CSV/Markdown 输出和 Mongo 沉淀
- 聚宽路线 A 已经落成单文件策略导出与执行约束
- `research/validation/` 和 `research/backtest_local/` 已经明确职责边界
- 模型训练、评估、回退机制已有基础骨架

但二期后仍然存在几个问题：

- 评分、标签、规则分散在 `entry_features`、`execution_engine`、`snapshot_ranking`、`joinquant_policy` 等位置
- 候选池状态没有被正式建模，当前仍偏“每日榜单”
- 因子有效性验证还没有成为规则进入主链路的门禁
- 产品化更多停留在输出文件和 Telegram 单票查询，还没有形成“每日监控闭环”
- 文档和代码中仍有部分已过期或未落地的入口说明

三期要解决的是：把现有分析能力组织成可持续迭代的策略系统。

## 2. 外部来源借鉴

### 2.1 Sequoia-X

Sequoia-X 值得借鉴的是“收盘后批处理选股器”的简洁形态：

- 数据同步
- 多策略扫描
- 结果合并
- 飞书推送
- 支持日常运行与历史回填

可借鉴内容：

- `RPS + 突破` 适合作为强势候选标签
- 海龟突破适合作为流动性与真实突破过滤
- 强势整理、涨停洗盘、错杀修复适合作为事件型观察标签
- 批处理后主动推送，适合形成每日候选池日报

不建议照搬内容：

- 不直接替换石论的数据层
- 不把策略类输出当作最终买入结论
- 不把每个策略继续写成互相平行的评分链

### 2.2 BaoStock 交易技术商城

交易技术商城值得借鉴的是“策略验证纪律”和“数据口径意识”。

可借鉴内容：

- 缠论二类买点裸策略回测显示，小级别形态单独使用优势有限
- 形态信号必须叠加市场、板块、长周期均线或 regime 过滤
- 前复权、手续费、滑点、无未来函数、逐根回测是策略可信度底线
- 分红稳定性可作为股东回报质量因子
- 业绩预告、利润增速、增长来源、行业价格变化可作为事件/基本面候选因子

不建议照搬内容：

- 不把单篇文章中的经验策略直接并入主策略
- 不把缠论形态信号当作独立买点
- 不在没有样本外验证前把分红、业绩预告等因子升为硬过滤

### 2.3 Shadowell/Alpha

Alpha 值得借鉴的是“规则中心 + 三池漏斗 + 产品工作台”。

可借鉴内容：

- 原子规则组合，而不是每个策略单独写死
- 复合候选池、重点关注池、买入池的状态迁移
- 规则扫描、回测、模拟盘、通知、前端展示形成闭环
- AI/Agent 只通过工具读取真实数据，不直接编造分析
- 数据健康、降级、通知、模拟盘是产品化系统的一部分

不建议照搬内容：

- 不一次性引入大前端、Agent、模拟盘、预测模型等完整平台
- 不在策略收益证明不足时叠加过多 AI 解释层
- 不让产品化复杂度反过来拖累本地研究效率

## 3. 三期总目标

三期总目标是建立四个闭环：

1. 候选池闭环
   - 从每日榜单升级为可迁移、可降级、可追踪的候选池状态。

2. 规则中心闭环
   - 从散落的 `flag/score` 升级为原子规则、候选标签和因子组。

3. 验证门禁闭环
   - 任何准备进入主策略链的规则，都必须有本地验证路径。

4. 产品运营闭环
   - 每日扫描、状态变化、重点票、风险变化、执行建议可以被自动汇总和推送。

## 4. 三期非目标

三期暂不追求以下内容：

- 不建设完整 Web 工作台
- 不引入复杂 Agent 编排
- 不重写整个 `ShilunPipeline`
- 不直接替换 Tushare/Mongo 数据底座
- 不把所有外部策略一次性实现
- 不把聚宽作为因子研究主战场

三期更强调：先把策略迭代链路变稳，再做界面和智能化。

## 5. 目标主链路

三期主链路统一为：

`universe -> data_quality -> features -> factor_groups -> atomic_rules -> candidate_pools -> validation -> ranking -> portfolio -> execution/backtest -> notification`

各层职责：

- `universe`：股票池、ST、停牌、流动性、上市时间、行业范围
- `data_quality`：复权口径、缺失值、价格连续性、异常成交量
- `features`：基础行情、趋势、波动、量能、结构、筹码
- `factor_groups`：趋势、结构、量价、筹码、风险、环境、事件、质量
- `atomic_rules`：可配置、可解释、可验证的最小规则单元
- `candidate_pools`：复合候选池、重点关注池、买入池、淘汰池
- `validation`：分组收益、RankIC、样本外、walk-forward、分市场阶段
- `ranking`：候选排序，不再混入解释层
- `portfolio`：仓位、持仓数、行业集中度、环境降仓
- `execution/backtest`：本地研究回测与聚宽执行回测
- `notification`：日报、风险提示、候选池变动、重点票追踪

## 6. 三期实施主线

### M1：因子池与规则中心

目标：

- 建立统一因子池清单
- 把外部策略思想转换为候选标签
- 停止继续增加无归属的零散字段

P0 任务：

- 建立因子池文档，分为趋势、结构、量价、筹码、风险、环境、事件、质量八组
- 梳理当前 `snapshot` 字段归属，标记为保留、观察、淘汰
- 定义原子规则接口：输入字段、输出标签、解释、验证路径
- 先设计 5 个候选标签：
  - `rps_breakout`
  - `turtle_breakout`
  - `high_tight_flag`
  - `earnings_surprise`
  - `dividend_quality`

P1 任务：

- 将候选标签接入 `SnapshotJob` 结果，但不直接改变主排序
- 为每个标签定义验证口径和未来收益观察窗口
- 建立规则配置文件或注册表，避免继续在多个模块里硬编码

完成标志：

- 因子池文档完成
- 第一批候选标签定义完成
- 每个候选标签都有验证路径

### M2：验证门禁

目标：

- 把 `research/validation/` 从目录说明升级为可运行验证入口
- 让规则先被证明，再进入排序、仓位或执行链

P0 任务：

- 建立单因子分组收益脚本
- 建立 RankIC / IC 分析脚本
- 建立按市场阶段、行业、趋势阶段分组的验证输出
- 统一验证输入，优先读取 `market_snapshot_records`
- 明确前复权、手续费、滑点、无未来函数口径

P1 任务：

- 建立候选标签验证报告
- 建立样本内、样本外拆分规则
- 建立 walk-forward 验证模板
- 建立因子保留、观察、淘汰的决策标准

完成标志：

- 至少 5 个候选标签有验证报告
- 至少 1 份因子有效性报告落盘
- 新规则进入主链路前必须引用验证报告

### M3：候选池状态系统

目标：

- 从“每日排名表”升级到“候选池状态迁移”
- 借鉴 Alpha 的三池漏斗，但保持石论轻量化

候选池定义：

- `candidate_pool`：复合候选池，满足基础质量、流动性、趋势或事件标签
- `watch_pool`：重点关注池，候选标签持续、结构改善、风险可控
- `buy_pool`：买入池，执行分、风险、环境、入场区都达标
- `reject_pool`：淘汰池，失效、风险过高、假突破、流动性不足

P0 任务：

- 设计候选池状态字段
- 设计升池、降池、淘汰、过期规则
- 在 Mongo 中沉淀每日池状态
- 输出每日状态变化报告

P1 任务：

- 支持连续 N 日跟踪同一股票的状态变化
- 支持重点关注池风险提醒
- 支持买入池和聚宽目标仓位对齐

完成标志：

- 每日全市场扫描可以输出四类池
- 每只股票可以追踪最近状态、状态原因、进入日期、过期日期
- 状态变化可以进入日报或 Telegram 推送

### M4：组合与本地回测闭环

目标：

- 把本地回测作为三期收益证明主战场
- 让 `portfolio` 真正接入主链路

P0 任务：

- 建立本地组合回测入口
- 接入候选池、排序、仓位、调仓频率
- 接入交易成本、滑点、涨跌停、停牌、新股、成交额约束
- 接入 `invalidation_level` 驱动的止损或减仓

P1 任务：

- 输出收益率、回撤、Sharpe、alpha、换手率、胜率、行业暴露
- 对比不同候选标签组合的收益贡献
- 对比不同市场环境下的策略表现

完成标志：

- `research/backtest_local/` 有可运行组合回测脚本
- 可以用历史 `market_snapshot_records` 回放策略
- 能输出一份阶段性收益证明报告

### M5：代码结构收敛

目标：

- 延续二期代码减法原则
- 让核心职责更清楚，减少评分和字段重复

P0 任务：

- 拆分 `ShilunPipeline` 中的数据上下文构造和 snapshot 组装逻辑
- 将 `entry_features` 输出归并到因子组，不再无限增加 `flag`
- 收敛 `execution_engine`，保留执行判断，不承担组合排序职责
- 将 `snapshot_ranking` 与 `portfolio/ranker` 的职责合并或明确边界
- 让 `SnapshotJob` 只做批处理编排，减少展示、研究、排序逻辑堆积

P1 任务：

- 修正 README 中已不存在的 `docs/beginner_rewrite_plan.md` 和 `shilun/simple/` 说明
- 建立“新增字段必须归属因子组”的文档规则
- 建立“新增规则必须有验证路径”的代码审查规则

完成标志：

- 主链路职责图更新
- 重复评分职责完成第二轮收敛
- 新增标签不再散落在多个模块中硬编码

### M6：轻量产品化闭环

目标：

- 不做大平台，但让每日使用体验完整
- 把候选池、重点票、风险变化、验证结果推送出来

P0 任务：

- 建立日报模板：
  - 全市场扫描概况
  - 新进入候选池
  - 升入重点关注池
  - 升入买入池
  - 风险降级或淘汰
  - Top N 重点票
  - 市场/板块环境摘要
- 抽象 notifier：
  - Telegram
  - Feishu 预留
- 把单票解释和每日摘要分开

P1 任务：

- 建立本地静态 HTML 或 Markdown 报告
- 支持按行业、候选标签、池状态查看
- 为后续 Web 控制台保留数据接口，但三期不做完整前端

完成标志：

- 每日扫描后能自动生成候选池日报
- 日报可通过 Telegram 发送
- 报告能解释“为什么进入池、为什么降级、下一步看什么”

## 7. 优先清理对象

### 7.1 `shilun/pipeline.py`

问题：

- 文件过长，同时承担数据加载、外部上下文、特征、结构、模型、解释组装

三期目标：

- 保留 `ShilunPipeline` 作为主入口
- 将外部上下文构造、snapshot 组装、证据文本构造逐步拆出

### 7.2 `shilun/features/entry_features.py`

问题：

- 同时输出位置、量价、风险、买点、阶段等大量字段

三期目标：

- 按因子组归并输出
- 新增字段必须说明属于哪个因子组
- 候选标签优先从规则中心生成，不继续直接散落在这里

### 7.3 `shilun/decision/execution_engine.py`

问题：

- 同时承担执行分、风险分、仓位、部分排序语义

三期目标：

- 保留执行建议和风险动作
- 将排序权重交给 `ranker`
- 将组合仓位交给 `portfolio`

### 7.4 `shilun/jobs/snapshot_job.py`

问题：

- 同时承担扫描编排、缓存、行业上下文、未来标签、导出、Mongo 写入

三期目标：

- 保留批处理编排
- 将候选池状态、验证标签、报告输出逐步外移

### 7.5 `snapshot_ranking` 与 `portfolio/ranker`

问题：

- 两者都与排序有关，但职责边界仍不够清楚

三期目标：

- `snapshot_ranking` 只做榜单展示字段转换
- `portfolio/ranker` 负责策略排序
- 或者在三期中合并成统一 `ranking` 模块

## 9. 推荐实施顺序

### 第 1 周：先定规则和字段

- 建立因子池清单
- 建立原子规则清单
- 梳理现有 `snapshot` 字段归属
- 确定第一批候选标签

### 第 2 周：先做验证，不急着改排序

- 建立 `research/validation/` 第一批脚本
- 对现有字段和第一批候选标签做分组收益验证
- 输出第一版保留、观察、淘汰清单

### 第 3 周：建立候选池状态

- 设计候选池 schema
- 将 `SnapshotJob` 输出接入候选池状态
- 生成每日候选池变化报告

### 第 4 周：接组合和日报

- 将候选池接入本地组合回测
- 输出第一版收益证明
- 抽象 Telegram/Feishu notifier
- 生成第一版每日策略报告

### 第 5-6 周：做代码收敛

- 收敛 `entry_features`
- 收敛 `execution_engine`
- 收敛 `snapshot_job`
- 修正文档漂移
- 固化新增规则的验证门禁

## 10. 三期验收标准

三期验收需要满足：

- 至少 5 个候选标签被定义并完成验证
- 至少 1 套候选池状态可每日更新
- 至少 1 个本地组合回测入口可运行
- 至少 1 份阶段性收益证明报告生成
- 每日扫描可以输出候选池变化摘要
- 新增规则必须能追溯到因子组和验证报告
- 聚宽单文件策略继续可导出、可运行
- 核心代码职责比二期更清晰，而不是更分散

## 29. Daily Push 真实状态修正（2026-05-18）

- 新行为改为 fail-fast：非 `--dry-run` 模式下没有真实通道会直接报错，提示配置飞书或 Telegram。
- `--dry-run` 继续用于本地预览日报文本，不代表推送成功。

下一步如果要让“每日自动推送”真正产品化，应补齐部署层：定时任务、环境变量检查、通道连通性自检和推送结果落库。

## 30. Daily Push 飞书优先收敛（2026-05-19）

本次根据当前使用决策，先把日报真实推送收敛为飞书优先：

- `.env` 已配置 `SHILUN_FEISHU_WEBHOOK_URL`，因此当前真实推送通道先使用飞书。
- Telegram 虽有 bot token，但没有 `SHILUN_TELEGRAM_PUSH_CHAT_IDS` 和公网 webhook base，暂不作为日报通道。
- `DailyPushJob` 不再通过 `getUpdates` 自动发现 Telegram chat id，避免用户只是配置了 bot token 时误发 Telegram。
- 后续如需恢复 Telegram 日报推送，必须显式配置 `SHILUN_TELEGRAM_PUSH_CHAT_IDS`。

## 32. Mongo 数据链路和定时同步（2026-05-19）

本次把页面卡住背后的数据链路补成更明确的闭环：

- `.env` 新增 `SHILUN_MONGO_URI=mongodb://localhost:27017` 和 `SHILUN_MONGO_DB=shilun`。
- 修正 Mongo 窄接口，使 `raw_market`、`market_snapshots`、`candidate_pools` 在真实 Mongo 路径下可用。
- `TushareSyncJob` 新增 `--latest-only` 和 `--skip-if-exists`，用于每日只同步最新开放交易日。
- `TushareSyncJob` 新增 `--history-year`，用于导入最近一年历史行情。
- Mongo upsert 改为支持 `bulk_write` 批量写入，避免最近一年历史导入时逐条 `update_one` 过慢。
- Web 控制台新增数据链路区：先查 Mongo 是否已有目标日期数据；有数据提示已更新；没有数据时可手动同步 Tushare 或切换到最近历史日期分析。
- 已创建 Codex 定时任务 `同步 Tushare 最新交易日到 Mongo`，每天 05:00 和 07:00 Asia/Shanghai 执行最新交易日同步。

链路原则：

1. Mongo 有目标日期数据：页面直接提示“已更新”，单票分析读 Mongo。
2. Mongo 没有目标日期数据：用户可点击同步最新交易日，显式调用 Tushare 后写 Mongo。
3. 如果不想调 Tushare：用户可切换到 Mongo 最近历史日期运行分析。
