# 石论交易决策系统 - 四期技术方案

本文档承接《08_三期迭代》，用于定义石论四期的技术方向、架构边界和实施路径。

三期已经把石论从“单票分析 + 批量榜单”推进到：

> Mongo-first 数据链路、策略 ID/版本、候选池状态、收益验证、日报推送和轻量 Web 控制台。

四期不再以继续堆功能为主，而是进入：

> 架构内核重构期。

核心方向是：用 Rust 或其他强类型语言承接稳定系统内核，用 Python 保留策略、因子、模型和研究效率。

---

## 1. 四期背景

当前系统已经具备可用闭环：

- Tushare 同步到 Mongo
- Mongo 原始行情驱动单票分析
- SnapshotJob 生成全市场榜单
- CandidatePoolJob 生成候选池状态
- DailyPushJob 生成日报并支持飞书推送
- `strategy_id + strategy_version` 用于策略收益验证
- Web 控制台可访问单票分析、数据状态、日报预览和推送

但 graphify 当前仍显示几个高连接节点：

- `MongoSnapshotStore`
- `ShilunPipeline`
- `DailyPushJob`
- `AppConfig`
- `SnapshotJob`

这说明复杂度已经不主要来自某个策略，而来自系统边界：

- 数据层、任务层、推送层、API 层和策略层仍在 Python 内互相引用
- Mongo store 承担连接、索引、原始数据、快照、候选池、兼容包装等多种职责
- 任务运行缺少稳定任务状态、日志、重试和进度模型
- 页面已经能点，但长任务仍不适合直接绑在 HTTP 请求里
- Python 同时承担系统编排和策略研究，长期会让核心链路越来越脆

四期要解决的是：把“系统内核”和“策略实现”分开。

---

## 2. 四期总目标

四期总目标是建立一个双语言、强边界、可观测的交易决策系统。

目标形态：

```text
shilun-core/        Rust 或其他强类型语言：系统内核、任务、数据、API 编排
shilun/             Python：策略、因子、模型、解释、兼容入口
research/           Python：验证、回测、训练、实验
web/                控制台页面，后续可独立前端化
docs/               数据契约、策略契约、运行手册
```

职责划分：

- Rust 负责稳定性
- Python 负责策略迭代速度
- Mongo 负责事实数据和任务状态
- Web 控制台负责可视化操作
- 文档和 schema 负责跨语言契约

四期不是“把 Python 全部重写成 Rust”，而是先做 sidecar 内核，再逐步迁移高稳定性模块。

---

## 3. 设计原则

### 3.1 系统内核强类型

以下内容应逐步从 Python 迁出：

- 任务调度
- 任务状态
- Mongo 数据访问
- 数据质量检查
- 同步进度
- 推送状态
- API 编排
- 长任务队列

这些模块变化频率低，但要求稳定、可观测、可恢复，适合 Rust。

### 3.2 策略层保持 Python

以下内容继续留在 Python：

- 因子计算
- 策略定义
- 候选标签实验
- 模型训练
- 本地回测
- LLM 解释
- 研究脚本

这些模块需要高迭代速度，Python 更适合。

### 3.3 先契约，后迁移

Rust 和 Python 之间必须先定义契约：

- 行情输入 schema
- 单票分析请求 schema
- 策略输出 schema
- 候选池状态 schema
- 任务状态 schema
- 推送结果 schema

没有契约前，不迁移核心逻辑。

### 3.4 不做一次性大重写

四期采用 strangler fig 模式：

1. 新能力先进 Rust sidecar
2. Python 旧能力继续可用
3. 单个边界稳定后再迁移调用方
4. 迁移完成后删除 Python 旧入口

## 4. 目标架构

### 4.1 总体链路

```text
Tushare
  |
  v
shilun-core data service
  |
  v
Mongo raw market data
  |
  v
shilun-core task orchestrator
  |
  +--> Python strategy worker
  |      |
  |      v
  |   strategy signals / analysis payload
  |
  v
Mongo snapshots / candidate pools / task runs
  |
  v
Web console / Feishu / Research validation
```

### 4.2 Rust 内核模块

建议新建 `shilun-core/`：

```text
shilun-core/
  Cargo.toml
  src/
    main.rs
    config.rs
    api/
    data/
    jobs/
    scheduler/
    contracts/
    telemetry/
```

模块职责：

- `config`
  - 读取 `.env`
  - 管理 Mongo、Tushare、飞书、Python worker 地址
- `data`
  - Mongo repository
  - 原始行情查询
  - 快照查询
  - 候选池查询
  - push/task run 查询
- `jobs`
  - 同步任务
  - snapshot 任务
  - candidate pool 任务
  - daily push 任务
- `scheduler`
  - 每日 05:00 / 07:00 同步
  - 收盘后 snapshot
  - 候选池生成
  - 日报推送
- `contracts`
  - JSON Schema / OpenAPI 类型
  - Python worker 输入输出结构
- `telemetry`
  - 日志
  - 任务进度
  - 错误追踪
  - 运行指标

### 4.3 Python 策略 worker

Python 保留为策略执行层：

```text
shilun/
  strategies/
  rules/
  pipeline.py
  models/
  llm_bridge/
  strategy_worker.py
```

`strategy_worker.py` 对 Rust 暴露稳定接口：

- `POST /strategy/analyze`
- `POST /strategy/batch-analyze`
- `POST /strategy/signals`
- `POST /strategy/explain`

输入来自 Rust 的标准 JSON，不直接读 Mongo。

## 5. 语言边界方案

四期可选三种调用方式。

### 5.1 第一阶段：HTTP sidecar

推荐起步方案。

Rust 和 Python 各自启动服务，通过 HTTP/JSON 通信。

优点：

- 边界清楚
- 调试简单
- 不绑定 Python ABI
- 方便页面和任务服务复用

缺点：

- 有序列化开销
- 本地部署多一个进程

适用阶段：

- M1-M3。

### 5.2 第二阶段：gRPC 或 Protobuf

当数据量和调用频率提高后，可迁移到 Protobuf。

优点：

- 强 schema
- 更高性能
- 适合批量分析

缺点：

- 开发复杂度更高
- Python 端调试略重

适用阶段：

- 全市场扫描变成稳定生产任务后。

### 5.3 第三阶段：PyO3

把 Rust 编译成 Python 扩展。

优点：

- Python 调用体验好
- 性能高

缺点：

- ABI 和构建链路复杂
- 不适合先做系统边界

适用阶段：

- 单个性能瓶颈明确后，例如指标计算、批量排序、风险计算。

四期建议顺序：

```text
HTTP/JSON sidecar -> Protobuf/gRPC -> 局部 PyO3
```

### 5.4 当前落地状态

当前已新增 `shilun-core/` 作为 Rust 内核骨架，先承接稳定状态机和跨语言契约，不直接替换 Python 生产链路。

已落地内容：

- `shilun-core/src/market_part1.rs`：PART1 大盘权限枚举、五维评分结构和权限分类函数。
- `docs/contracts/market_part1.schema.json`：Python API 与未来 Rust sidecar 的 JSON 输出契约。
- `docs/rustization_status.md`：Rust 化边界、已完成项和下一步迁移顺序。
- `shilun/market/part1.py`：当前生产可用的 Python 版 PART1 实现。

当前原则：

```text
Python 先负责生产可用，Rust 先负责强类型内核和契约稳定。
等状态机、任务状态和数据契约稳定后，再迁移 Mongo repository 与 HTTP sidecar。
```

## 8. 四期实施路线

### M1：契约和任务状态底座

目标：

- 不迁移复杂逻辑，先把任务状态模型立住。

交付：

- `docs/contracts/` 第一版 schema
- Mongo 新增 `task_runs`、`push_runs`
- Python 现有任务写入任务状态
- Web 控制台显示任务状态
- Daily push 记录推送结果

验收：

- 页面能看到今天数据同步是否成功
- 页面能看到 snapshot 是否生成
- 页面能看到飞书是否推送成功
- 失败原因能落库

### M2：Rust sidecar 骨架

目标：

- 新建 `shilun-core`，先做只读能力和任务状态 API。

交付：

- Rust 项目初始化
- 读取 `.env`
- 连接 Mongo
- 查询数据状态
- 查询任务状态
- 暴露 `/health`、`/data/status`、`/tasks`

验收：

- Rust 服务能独立启动
- Web 控制台可切到 Rust 状态 API
- Python 原有链路不受影响

### M3：数据同步迁移到 Rust

目标：

- 把 Tushare -> Mongo 的稳定同步链路迁到 Rust。

交付：

- 最新交易日同步
- 最近一年历史同步
- 断点续跑
- 单日失败不中断
- 同步进度写 `task_runs`
- 数据质量检查写 `data_quality_reports`

验收：

- 每日 05:00 / 07:00 同步由 Rust 调度
- Mongo 有数据时不重复拉 Tushare
- Tushare 超时时能记录失败日期并续跑

### M4：任务编排迁移到 Rust

目标：

- Rust 管理 snapshot、candidate pool、daily push 的运行状态。

交付：

- Rust 创建任务
- Python worker 执行策略分析
- Rust 收集结果并落库
- Web 控制台查看进度
- 飞书推送状态落库

验收：

- 页面点击 snapshot 不阻塞
- 页面能看到进度
- 失败可重试
- 推送结果可追踪

### M5：策略 worker 标准化

目标：

- Python 策略层从系统编排中解耦。

交付：

- `strategy_worker.py`
- 标准 `AnalysisRequest`
- 标准 `AnalysisResult`
- 批量分析接口
- 策略 ID/版本入参
- 策略信号输出契约

验收：

- Rust 不需要 import Python 内部模块
- Python 不需要直接读 Mongo
- 新策略只需要注册到 Python 策略层

### M6：页面控制台升级

目标：

- 从当前单页操作台升级为可运营控制台。

交付页面：

- 首页仪表盘
- 数据同步页
- 单票分析页
- 全市场快照页
- 候选池页
- 策略验证页
- 推送记录页
- 系统日志页

验收：

- 非命令行用户可以完成日常操作
- 每一步都有状态和错误提示
- 关键结果可追溯

## 12. 风险和约束

### 12.1 Rust 重写风险

风险：

- 一次性重写会拖慢策略迭代
- Rust 和 Python 双栈增加部署复杂度
- 数据契约不稳时迁移会反复返工

控制方式：

- 先 sidecar，不替换 Python
- 先只读和任务状态，再迁移写路径
- 每迁移一块都保留回滚入口

### 12.2 策略表达受限风险

风险：

- 如果策略逻辑过早迁到 Rust，会降低策略调整速度

控制方式：

- 策略继续 Python
- Rust 只管输入、输出、状态和调度
- 策略输出必须结构化

### 12.3 数据一致性风险

风险：

- Rust 和 Python 同时写 Mongo 可能产生状态不一致

控制方式：

- 明确写权限
- 原始数据只由数据同步层写
- 策略结果只由任务编排层写
- Python worker 返回结果，不直接落库

### 12.4 长任务体验风险

风险：

- 页面直接请求长任务会超时

控制方式：

- 页面只创建任务
- 后端异步执行
- 页面轮询 task status

## 13. 四期验收标准

四期完成后，应满足：

1. 数据链路可观测
   - 今日是否同步、同步多少条、失败哪些日期，一眼可见。

2. 任务链路可追踪
   - snapshot、候选池、推送、验证都有 task run。

3. Python 策略层可插拔
   - 新策略不需要改系统调度，只需要注册策略 ID 和版本。

4. Rust 内核可独立运行
   - 即使 Python worker 暂时不可用，数据状态、任务状态、推送记录仍可查询。

5. Web 控制台可运营
   - 用户不依赖命令行也能完成日常操作。

6. 旧 Python god nodes 明显下降
   - `MongoSnapshotStore`、`DailyPushJob`、`SnapshotJob` 不再承担系统级编排。

---

## 14. 第一阶段建议

四期第一阶段不建议立刻写大量 Rust 业务逻辑。

建议先做：

1. 新建 `docs/contracts/`
2. 新建 `task_runs` 和 `push_runs`
3. 让现有 Python 任务先写任务状态
4. Web 控制台显示任务状态
5. 新建 `shilun-core` Rust 空壳，只做 `/health` 和 Mongo 只读状态

这样能先把系统边界立起来，而不是掉进重写泥潭。

---

## 15. 一句话原则

四期不是把石论改成 Rust 项目，而是把石论升级成：

> Rust 管稳定系统，Python 管策略智慧，Mongo 管事实记录，页面管日常运营。
